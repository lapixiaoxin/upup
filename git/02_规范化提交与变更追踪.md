# 规范化提交与变更追踪 (Commit Conventions & Tracking)

在企业级协作中，Git 的历史记录不仅仅是代码的备份，它更是**项目的演进史、业务的变更单，以及排查线上故障时的第一手线索**。如果团队的提交记录充斥着 `update`、`fix bug`、`ok` 这种无意义的词汇，一旦出问题，回溯的成本将是灾难性的。

## 1. 约定式提交 (Conventional Commits)

为了解决提交信息混乱的问题，目前业界最通用的标准是基于 Angular 团队规范演化而来的 **约定式提交 (Conventional Commits)**。

### 1.1 标准格式
一条标准的 Commit Message 应该包含三个部分：Header、Body 和 Footer。通常我们在日常开发中严格要求的是 **Header**：

```text
<type>(<scope>): <subject>
// 空一行
<body>
// 空一行
<footer>
```

### 1.2 核心的 Type 解析
在 Header 中，最关键的是 `<type>`（类型），它直接说明了这次提交的目的。企业中常用的 Type 如下：

*   `feat`: 新增功能 (Feature)
*   `fix`: 修复 Bug
*   `docs`: 仅仅修改了文档 (Documentation)
*   `style`: 代码格式修改，不影响逻辑 (如去掉空格、改变缩进、增加分号等)
*   `refactor`: 代码重构 (既不是新增功能，也不是修改 bug 的代码变动)
*   `perf`: 优化相关，比如提升性能、体验
*   `test`: 增加测试用例或修改测试用例
*   `build`: 影响构建系统或外部依赖的更改 (如 glup, broccoli, npm)
*   `ci`: 修改 CI 配置文件和脚本 (如 GitHub Actions, GitLab CI, Jenkins)
*   `chore`: 其他不修改 `src` 或测试文件的杂项更改 (如修改 `.gitignore`)
*   `revert`: 回滚到上一个版本

### 1.3 Scope (作用域) 与 Subject (简述)
*   **Scope (可选)**：用于说明 commit 影响的范围，比如 `(user)`、`(payment)`、`(router)`、`(components)`。这对于单体仓库 (Monorepo) 或者大项目非常有用。
*   **Subject (必填)**：本次提交的简短描述。规范要求：动词开头、第一人称现在时、不要首字母大写、结尾不要加句号。

**标准示例：**
> `feat(payment): 支持微信扫码支付`
> `fix(user): 修复登录接口超时无提示的问题`

## 2. 业务关联：与任务管理系统打通

在成熟的企业工程中，代码必须与业务需求/缺陷单关联起来。这被称为**变更追踪 (Traceability)**。

通常，我们会将工单号 (Jira ID, 飞书任务 ID, Trello Card ID) 融入到 Commit 规范中。

**最佳实践格式：**
> `feat(cart): [JIRA-4096] 购物车支持批量删除商品`

**这样做的巨大优势：**
1.  **双向追溯：** 在 GitLab/GitHub 上看代码时，能直接知道这行代码是因为哪个需求写的；在 Jira 里看需求时，能直接看到开发提交了哪些代码。
2.  **自动化周报/发版说明：** 系统可以通过提取提交信息里的工单号，自动生成本周完成了哪些业务需求。

## 3. 自动化版本号与 CHANGELOG

遵循了“约定式提交”后，Git 的历史记录就变成了机器可读的结构化数据。这就引出了现代工程化的一个神级应用：**自动发版与生成 CHANGELOG**。

### 3.1 语义化版本控制 (SemVer)
软件版本号通常格式为 `主版本号.次版本号.修订号` (如 `v1.4.2`)。结合规范提交，可以实现机器自动决定版本升级：
*   只要提交里有 `feat` -> 自动升级次版本号 (Minor)，变成 `v1.5.0`
*   只要提交里只有 `fix` -> 自动升级修订号 (Patch)，变成 `v1.4.3`
*   如果提交的 Footer 包含 `BREAKING CHANGE` (破坏性更新) -> 自动升级主版本号 (Major)，变成 `v2.0.0`

### 3.2 自动生成更新日志
工具（如 `standard-version` 或 `semantic-release`）会扫描上一个 Tag 到现在的提交记录，自动将所有的 `feat` 和 `fix` 提取出来，生成一份漂亮的 `CHANGELOG.md`，彻底解放双手。

## 4. 企业级工具链落地 (如何强制约束？)

光有规范是不够的，靠程序员自觉是不靠谱的。企业工程必须通过底层工具进行**硬性拦截**。前端/Node.js 生态中最经典的组合是 `Husky` + `Commitlint` + `Commitizen`。

### 4.1 Commitizen (交互式提交)
开发者可能记不住那么多 `type`。通过引入 `commitizen`（通常配合 `cz-customizable`），开发者只需要敲入 `npm run commit` 或 `git cz`，终端就会出现一个友好的**交互式菜单**，引导你一步步选择 Type、输入 Scope、填写 Jira ID，最终自动拼接成标准格式。

### 4.2 Husky (Git Hooks 底层拦截)
`Husky` 是一个让你更容易使用 Git Hooks 的工具。我们通常会用到 `commit-msg` 这个钩子。
它的作用是：**在开发者敲下回车键试图提交的那一瞬间，拦截这个动作**。

### 4.3 Commitlint (校验规则)
被 `Husky` 拦截下来的 commit 信息，会交给 `commitlint` 去校验。
如果格式不符合 `<type>(<scope>): <subject>`，或者没有携带 `[JIRA-xxx]` 工单号，`commitlint` 会直接报错并在终端输出红字，**导致 Commit 失败**。

**拦截效果演示：**
```bash
$ git commit -m "修复了一个大 bug"

⧗   input: 修复了一个大 bug
✖   subject may not be empty [subject-empty]
✖   type may not be empty [type-empty]

✖   found 2 problems, 0 warnings
ⓘ   Get help: https://github.com/conventional-changelog/commitlint/#what-is-commitlint

# 提交失败！代码仍在本地工作区。
```

## 5. 总结
“规范化提交”是前端工程化、CI/CD 自动化流水线的第一步。它不仅让代码历史赏心悦目，更是实现需求追踪、自动发版的基石。在企业项目中，务必尽早引入 `Husky + Commitlint` 进行强制约束。
