# 知识库全局索引与协作规范

本仓库用于沉淀跨领域的系统化学习内容。它不是临时笔记集合，而是面向长期复用、持续扩展、可公开发布的知识管理仓库。

顶层 `AGENTS.md` 负责定义全局知识组织规则、写作规范和版块入口。每个具体知识版块在自己的目录下维护 `.index.md`，用于说明该领域的学习路径、内容边界、文档清单和维护约束。

## 1. 仓库定位

* **长期知识资产**：内容应能被反复查阅、迭代和迁移，而不是只解决一次性问题。
* **实战优先**：技术类内容优先面向企业工程实践；非技术类内容优先面向真实决策场景。
* **结构化学习**：每个领域需要形成从基础概念、核心模型、实践方法到风险边界的递进路径。
* **模块化扩展**：Git、Docker、个人理财等主题应作为独立版块管理，避免所有内容堆叠在单一文档中。
* **公开友好**：内容默认会被上传到 GitHub 仓库，因此需要保持清晰、克制、可读、可维护。

## 4. 版块目录约定

每个一级知识版块必须遵守以下结构约定：

* **必须维护 `.index.md`**：作为当前版块的入口，说明版块定位、学习路径、文档清单、内容边界和后续规划。
* **复杂主题单独成文**：每个独立且复杂的知识点拆成单独 `.md` 文件，不在 `.index.md` 中堆叠长篇正文。
* **大纲文件使用 `00_` 前缀**：如 `00_Docker企业级容器化实战大纲.md`，用于承载该领域的全景图。
* **专题文件使用数字编号**：如 `01_镜像构建与Dockerfile规范.md`，保证阅读顺序稳定。
* **新增、删除、重命名文档后同步更新索引**：目录结构和 `.index.md` 必须保持一致。

## 5. 内容写作规范

### 5.1 结构原则

* **Why -> What -> How**：先说明为什么需要，再解释是什么，最后给出怎么做。
* **先模型，后细节**：先建立知识框架，再进入命令、配置、案例和排障。
* **先通用，再企业化**：先讲通用原则，再讲企业流程、团队协作、权限、安全、审计、成本等现实问题。
* **对比与取舍**：涉及多种方案时，必须说明优缺点、适用场景和不适用场景。

### 5.2 表达原则

* **高知识密度**：避免空泛铺垫，直接给出关键概念、判断标准和落地方式。
* **客观专业**：不做情绪化表达，不夸大结论，不制造确定性幻觉。
* **必要时使用引用块**：用 `>` 标注注意事项、最佳实践、风险提示或边界条件。
* **绝对不使用 Emoji**：不使用表情符号或无实际意义的装饰性符号。

### 5.3 技术类内容要求

* 明确适用环境，如本地开发、测试环境、生产环境、CI/CD、团队协作等。
* 命令示例必须给出上下文，避免只罗列命令。
* 企业实践必须覆盖权限、安全、回滚、审计、自动化和协作成本。
* 涉及工具版本、配置差异或平台差异时，需要标注前提条件。

### 5.4 理财类内容要求

* 内容仅用于知识学习和方法论整理，不构成投资建议。
* 必须区分事实、假设、观点和行动建议。
* 涉及收益、风险、税务、保险、投资品时，需要说明适用条件和潜在风险。
* 不使用“稳赚”“必然”“无风险收益”等绝对化表述。

## 6. Agent 协作规则

当 AI Agent 在本仓库内新增或修改知识文档时，必须遵守以下流程：

1. 先阅读顶层 `AGENTS.md`，确认全局规范。
2. 再阅读目标版块的 `.index.md`，确认该领域的内容边界和学习路径。
3. 如果是新增领域，先创建领域目录和 `.index.md`。
4. 如果是新增复杂知识点，创建独立 `.md` 文件，并同步更新对应 `.index.md`。
5. 如果是修改既有内容，保持原有结构风格，不做无关重构。
6. 如果发现目录和索引不一致，优先修正索引。

## 7. 当前仓库 Git 维护约定

本仓库是个人知识管理项目，不采用企业多人协作下的严格分支流程。除非用户明确要求新建分支、发起 PR 或模拟企业协作流程，否则 Agent 可以直接在 `main` 分支上维护文档。

### 7.1 默认工作流

日常文档更新默认使用以下流程：

```bash
git status --branch --short
git pull --ff-only origin main
# 修改文档
git add <changed-files>
git commit -m "docs(scope): subject"
git push origin main
```

约束说明：

* 修改前先确认当前分支和工作区状态，避免覆盖用户未提交的改动。
* 推送前优先使用 `git pull --ff-only origin main` 同步远端，避免产生不必要的 merge commit。
* 默认不新建功能分支；只有在大规模重构、长期未完成内容、用户明确要求隔离修改时才创建分支。
* 默认不使用 `git push --force` 或 `git push --force-with-lease`。
* `.DS_Store` 等本地系统文件必须保持忽略状态，不得提交到仓库。

### 7.2 当前远程仓库

当前仓库远程地址使用个人 GitHub SSH Host 别名：

```bash
origin git@github.com-personal:lapixiaoxin/upup.git
```

如果后续 Agent 发现推送失败，应优先检查：

```bash
git remote -v
ssh -T github.com-personal
```

> `github.com-personal` 依赖本机 `~/.ssh/config` 中的 Host 配置，用于隔离个人 GitHub SSH Key，不应覆盖公司 SSH Key。

### 7.3 提交身份

本仓库属于个人知识管理项目，提交身份应优先使用个人 GitHub 账号信息，而不是公司全局 Git 配置。

检查当前仓库本地提交身份：

```bash
git config --local user.name
git config --local user.email
```

如未配置，应在当前仓库内设置本地身份：

```bash
git config user.name "lapixiaoxin"
git config user.email "<personal-email>"
```

> 以上配置只影响当前仓库，不影响公司项目的全局 Git 配置。

### 7.4 提交信息规范

本仓库以 Markdown 文档维护为主，提交信息采用轻量级约定式提交：

```text
<type>(<scope>): <subject>
```

常用 `type`：

| Type | 使用场景 |
| --- | --- |
| `docs` | 新增、修改、重构知识文档 |
| `chore` | 调整仓库配置、忽略文件、目录结构等非正文内容 |
| `fix` | 修正文档中的错误、失效链接、明显不准确表述 |
| `refactor` | 重组文档结构但不改变核心内容 |

常用 `scope`：

| Scope | 使用场景 |
| --- | --- |
| `global` | 顶层规范、全局索引、仓库结构 |
| `git` | Git 知识版块 |
| `docker` | Docker 知识版块 |
| `finance` | 个人理财知识版块 |

示例：

```bash
git commit -m "docs(global): 更新仓库 Git 维护约定"
git commit -m "docs(git): 补充分支管理学习路径"
git commit -m "chore(global): 忽略 macOS 本地系统文件"
git commit -m "fix(finance): 修正风险提示表述"
```

### 7.5 Agent 提交边界

Agent 修改本仓库时，应遵守以下提交边界：

* 只提交与当前用户请求直接相关的文件。
* 不把用户未要求修改的知识版块顺手重构。
* 不提交本地缓存、系统文件、编辑器临时文件或凭据文件。
* 如果发现工作区已有用户改动，必须保留并基于现状继续工作。
* 未经用户要求，不为普通文档更新创建分支或 PR。
